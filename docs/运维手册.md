## 1. 系统架构概览

### 1.1 服务组成

| 服务 | 镜像/技术栈 | 端口 | 说明 |
|------|------------|------|------|
| `postgres` | postgres:16-alpine | 5432（内部） | PostgreSQL 数据库 |
| `app` | Python 3.12-slim + FastAPI + Gunicorn | 8000 | Web 应用服务 |

### 1.2 数据卷

| 卷名 | 用途 | 宿主机路径 |
|------|------|-----------|
| `pg_data` | PostgreSQL 数据持久化 | Docker 管理 |
| `app_data` | 应用数据（上传文件等） | Docker 管理 |

### 1.3 关键配置

- **数据库名**: `it_ticketing`
- **数据库用户**: `ticketing`
- **数据库密码**: 通过 `.env` 文件中的 `DB_PASSWORD` 设置
- **健康检查端点**: `GET /health`
- **应用端口**: `8000`（可通过 `APP_PORT` 环境变量修改）

## 2. 日常运维任务

### 2.1 服务启动

```bash
# 进入项目目录
cd /path/to/it-ticketing

# 启动所有服务（后台运行）
docker compose up -d

# 启动并重新构建镜像（代码更新后使用）
docker compose up -d --build
```

### 2.2 服务停止

```bash
# 停止所有服务（保留数据卷）
docker compose down

# 停止所有服务并删除数据卷（⚠️ 数据将丢失）
docker compose down -v
```

### 2.3 服务重启

```bash
# 重启所有服务
docker compose restart

# 重启单个服务
docker compose restart app
docker compose restart postgres

# 停止后重新启动（推荐，确保配置生效）
docker compose down && docker compose up -d
```

### 2.4 查看服务状态

```bash
# 查看所有容器运行状态
docker compose ps

# 查看详细状态（含健康检查）
docker compose ps -a

# 查看容器资源使用情况
docker stats

# 通过健康检查端点验证服务
curl http://localhost:8000/health
# 返回: {"status":"ok"}
```

### 2.5 查看日志

```bash
# 查看所有服务日志（实时跟踪）
docker compose logs -f

# 仅查看 app 服务日志
docker compose logs -f app

# 仅查看 postgres 服务日志
docker compose logs -f postgres

# 查看最近 100 行日志
docker compose logs --tail=100 app

# 查看指定时间范围的日志
docker compose logs --since 2026-06-29T10:00:00 app
```

### 2.6 进入容器调试

```bash
# 进入 app 容器
docker compose exec app bash

# 进入 postgres 容器
docker compose exec postgres bash

# 在 postgres 容器中连接数据库
docker compose exec postgres psql -U ticketing -d it_ticketing
```

## 3. 数据库运维

### 3.1 数据库备份

#### 3.1.1 使用 pg_dump 备份（通过 Docker）

```bash
# 备份整个数据库（自定义格式，推荐）
docker compose exec -T postgres pg_dump -U ticketing -d it_ticketing -Fc > backup_$(date +%Y%m%d_%H%M%S).dump

# 备份整个数据库（SQL 文本格式，可直接阅读）
docker compose exec -T postgres pg_dump -U ticketing -d it_ticketing > backup_$(date +%Y%m%d_%H%M%S).sql

# 仅备份数据（不含表结构）
docker compose exec -T postgres pg_dump -U ticketing -d it_ticketing -Fc --data-only > backup_data_$(date +%Y%m%d_%H%M%S).dump

# 仅备份表结构
docker compose exec -T postgres pg_dump -U ticketing -d it_ticketing -Fc --schema-only > backup_schema_$(date +%Y%m%d_%H%M%S).dump
```

#### 3.1.2 备份脚本（推荐）

创建 `/opt/scripts/backup_it_ticketing.sh`：

```bash
#!/bin/bash
# IT工单系统数据库备份脚本

BACKUP_DIR="/opt/backups/it-ticketing"
RETENTION_DAYS=30
PROJECT_DIR="/path/to/it-ticketing"

mkdir -p "$BACKUP_DIR"

cd "$PROJECT_DIR"
FILENAME="it_ticketing_$(date +%Y%m%d_%H%M%S).dump"

docker compose exec -T postgres pg_dump -U ticketing -d it_ticketing -Fc > "$BACKUP_DIR/$FILENAME"

# 压缩备份
gzip "$BACKUP_DIR/$FILENAME"

# 清理旧备份（保留最近 30 天）
find "$BACKUP_DIR" -name "it_ticketing_*.dump.gz" -mtime +$RETENTION_DAYS -delete

echo "Backup completed: $FILENAME.gz"
```

### 3.2 数据库恢复

#### 3.2.1 使用 pg_restore 恢复

```bash
# 恢复自定义格式备份（.dump 文件）
docker compose exec -T postgres pg_restore -U ticketing -d it_ticketing --clean --if-exists < backup_20260629_120000.dump

# 恢复压缩备份
gunzip -c backup_20260629_120000.dump.gz | docker compose exec -T postgres pg_restore -U ticketing -d it_ticketing --clean --if-exists

# 恢复 SQL 文本格式备份
docker compose exec -T postgres psql -U ticketing -d it_ticketing < backup_20260629_120000.sql
```

#### 3.2.2 恢复前注意事项

1. **先备份当前数据**：恢复操作会覆盖现有数据，务必先备份
2. **停止应用服务**：建议先停止 app 服务，避免恢复期间的数据冲突
   ```bash
   docker compose stop app
   # 执行恢复...
   docker compose start app
   ```
3. **验证恢复结果**：恢复后检查关键表数据量
   ```bash
   docker compose exec postgres psql -U ticketing -d it_ticketing -c "SELECT count(*) FROM tickets;"
   docker compose exec postgres psql -U ticketing -d it_ticketing -c "SELECT count(*) FROM users;"
   ```

### 3.3 定时备份建议

#### 3.3.1 配置 Cron 定时任务

```bash
# 编辑 crontab
crontab -e

# 添加以下规则：

# 每天凌晨 2:00 执行备份
0 2 * * * /opt/scripts/backup_it_ticketing.sh >> /var/log/it-ticketing-backup.log 2>&1

# 每周日凌晨 3:00 执行完整备份并同步到远程
0 3 * * 0 /opt/scripts/backup_it_ticketing_full.sh >> /var/log/it-ticketing-full-backup.log 2>&1
```

#### 3.3.2 备份策略建议

| 备份类型 | 频率 | 保留周期 | 说明 |
|----------|------|----------|------|
| 每日增量 | 每天 | 7 天 | 快速恢复近期数据 |
| 每周全量 | 每周 | 30 天 | 完整备份，用于灾备 |
| 每月归档 | 每月 | 12 个月 | 长期归档，满足合规需求 |

#### 3.3.3 远程备份同步

```bash
# 同步到远程服务器（rsync）
rsync -avz /opt/backups/it-ticketing/ user@backup-server:/backups/it-ticketing/

# 同步到 AWS S3
aws s3 sync /opt/backups/it-ticketing/ s3://my-backup-bucket/it-ticketing/

# 同步到阿里云 OSS
ossutil cp -r /opt/backups/it-ticketing/ oss://my-backup-bucket/it-ticketing/
```

### 3.4 数据清理策略

#### 3.4.1 清理已关闭的工单（建议定期执行）

```sql
-- 查看 90 天前已关闭的工单数量
SELECT count(*) FROM tickets
WHERE status = 'closed' AND updated_at < NOW() - INTERVAL '90 days';

-- 归档已关闭工单到备份表（建议先创建归档表）
CREATE TABLE IF NOT EXISTS tickets_archive (LIKE tickets INCLUDING ALL);
INSERT INTO tickets_archive SELECT * FROM tickets
WHERE status = 'closed' AND updated_at < NOW() - INTERVAL '90 days';

-- 删除已归档的工单
DELETE FROM tickets
WHERE status = 'closed' AND updated_at < NOW() - INTERVAL '90 days';
```

#### 3.4.2 清理过期附件

```bash
# 查看附件使用空间
docker compose exec app du -sh app/static/uploads/

# 在数据库中查找孤立附件（工单已删除但附件仍在）
docker compose exec postgres psql -U ticketing -d it_ticketing -c "
  SELECT a.id, a.ticket_id, a.filename, a.created_at
  FROM attachments a
  LEFT JOIN tickets t ON a.ticket_id = t.id
  WHERE t.id IS NULL;
"
```

#### 3.4.3 清理过期密码重置令牌

```sql
-- 删除超过 24 小时的密码重置令牌
DELETE FROM password_resets WHERE created_at < NOW() - INTERVAL '24 hours';
```

#### 3.4.4 定期 VACUUM 维护

```sql
-- PostgreSQL 自动 VACUUM 通常已足够，但可以手动执行以回收空间
VACUUM ANALYZE;
VACUUM FULL;  -- ⚠️ 会锁表，建议在维护窗口执行
```

## 4. 故障排查

### 4.1 服务无法启动

#### 症状：`docker compose up -d` 后容器反复重启或处于 unhealthy 状态

**排查步骤：**

```bash
# 1. 查看容器状态
docker compose ps -a

# 2. 查看 app 服务日志
docker compose logs app --tail=50

# 3. 查看 postgres 服务日志
docker compose logs postgres --tail=50

# 4. 检查端口占用
lsof -i :8000
lsof -i :5432
```

**常见原因及解决方案：**

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `Error: Database is not ready` | PostgreSQL 未完成初始化 | 等待几秒后重试，检查 `docker compose logs postgres` |
| `Connection refused` | 数据库密码错误 | 检查 `.env` 文件中的 `DB_PASSWORD` |
| `Port 8000 already in use` | 端口被占用 | 修改 `.env` 中 `APP_PORT` 或释放端口 |
| `JWT_SECRET_KEY must be at least 32 characters` | JWT 密钥太短 | 在 `.env` 中设置至少 32 位的 `JWT_SECRET_KEY` |
| `Disk full` | 磁盘空间不足 | 清理磁盘空间（见 4.4 节） |
| 镜像构建失败 | 网络问题或依赖冲突 | 使用国内镜像源或检查 `requirements.txt` |

### 4.2 数据库连接失败

#### 症状：应用日志显示 `could not connect to server` 或 `Connection refused`

**排查步骤：**

```bash
# 1. 确认 postgres 容器正在运行
docker compose ps postgres

# 2. 检查 postgres 健康状态
docker inspect it-ticketing-postgres-1 | grep -A 5 Health

# 3. 测试数据库连接
docker compose exec postgres pg_isready -U ticketing -d it_ticketing

# 4. 测试应用到数据库的网络连通性
docker compose exec app python -c "
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.settimeout(5)
result = s.connect_ex(('postgres', 5432))
print('Connected' if result == 0 else f'Failed: {result}')
s.close()
"

# 5. 检查数据库进程
docker compose exec postgres ps aux | grep postgres
```

**解决方案：**

```bash
# 重启数据库服务
docker compose restart postgres

# 如果问题持续，重建数据库容器（⚠️ 保留数据卷）
docker compose down
docker compose up -d postgres
```

### 4.3 页面访问慢

**排查步骤：**

```bash
# 1. 检查系统资源使用
docker stats

# 2. 检查数据库慢查询
docker compose exec postgres psql -U ticketing -d it_ticketing -c "
  SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
  FROM pg_stat_activity
  WHERE (now() - pg_stat_activity.query_start) > interval '5 seconds'
    AND state != 'idle'
  ORDER BY duration DESC;
"

# 3. 检查活跃连接数
docker compose exec postgres psql -U ticketing -d it_ticketing -c "
  SELECT count(*) AS active_connections FROM pg_stat_activity WHERE state = 'active';
"

# 4. 检查 Worker 进程数
docker compose exec app ps aux | grep gunicorn | grep -v grep

# 5. 测试健康检查响应时间
time curl -s http://localhost:8000/health
```

**优化建议：**

| 问题 | 解决方案 |
|------|----------|
| CPU 使用率高 | 增加 `WORKERS` 环境变量（建议 = CPU 核数 × 2） |
| 内存不足 | 增加服务器内存，或减少 Worker 数 |
| 数据库查询慢 | 检查并添加索引，执行 `VACUUM ANALYZE` |
| 连接数过多 | 检查是否有连接泄漏，考虑使用连接池 |
| 静态资源加载慢 | 配置 Nginx 反向代理 + 静态资源缓存 |

### 4.4 磁盘空间不足

**排查步骤：**

```bash
# 1. 检查磁盘使用
df -h

# 2. 查看 Docker 占用空间
docker system df

# 3. 查看各卷占用空间
docker system df -v | grep -A 5 "Local Volumes"

# 4. 查看大文件
find /var/lib/docker -type f -size +100M | sort -rh | head -20
```

**清理方案：**

```bash
# 清理未使用的 Docker 资源（镜像、容器、网络、卷）
docker system prune -a

# 清理构建缓存
docker builder prune

# 清理日志文件
truncate -s 0 $(docker inspect --format='{{.LogPath}}' it-ticketing-app-1)
truncate -s 0 $(docker inspect --format='{{.LogPath}}' it-ticketing-postgres-1)

# 限制 Docker 日志大小（在 docker-compose.yml 中配置）
# 为每个服务添加：
#   logging:
#     driver: "json-file"
#     options:
#       max-size: "10m"
#       max-file: "3"
```

## 5. 升级指南

### 5.1 代码更新流程

#### 5.1.1 标准升级步骤

```bash
# 1. 进入项目目录
cd /path/to/it-ticketing

# 2. 备份数据库（务必执行！）
./backup.sh  # 或手动执行数据库备份命令

# 3. 拉取最新代码
git pull origin master

# 4. 查看变更内容
git log --oneline -10

# 5. 重新构建并启动服务
docker compose up -d --build

# 6. 验证服务状态
docker compose ps
curl http://localhost:8000/health

# 7. 查看日志确认无异常
docker compose logs app --tail=20
```

#### 5.1.2 零停机更新（滚动更新）

```bash
# 如果有多实例部署，可以逐个更新（需配合负载均衡器）
docker compose up -d --no-deps --build --scale app=3 app
# 逐个移除旧实例...
```

### 5.2 数据库迁移说明

本项目使用 SQLAlchemy 的 `Base.metadata.create_all` 自动创建表结构，同时在 `app/db_migrations.py` 中通过 `ensure_db_schema()` 处理增量迁移。

**迁移逻辑：**
- 启动时自动检查并创建缺失的表和列
- 不会删除或修改已有列
- 新增的列会使用默认值填充

**注意事项：**
- 升级前务必备份数据库
- 如果涉及复杂的数据结构变更，建议在维护窗口执行
- 首次启动或迁移失败时检查应用日志：
  ```bash
  docker compose logs app | grep -i "migration\|schema"
  ```

### 5.3 回滚方案

#### 5.3.1 代码回滚

```bash
# 1. 回退代码到指定版本
git log --oneline -20  # 找到要回退的 commit hash
git checkout <commit-hash>

# 2. 重新构建
docker compose up -d --build

# 3. 验证
curl http://localhost:8000/health
```

#### 5.3.2 数据库回滚

```bash
# 恢复之前备份的数据库
docker compose stop app

# 恢复备份
docker compose exec -T postgres pg_restore -U ticketing -d it_ticketing --clean --if-exists < backup_before_upgrade.dump

docker compose start app
```

#### 5.3.3 完整回滚脚本

```bash
#!/bin/bash
# 紧急回滚脚本

echo "⚠️ 开始回滚操作..."

# 停止应用
docker compose stop app

# 恢复数据库
echo "恢复数据库备份..."
docker compose exec -T postgres pg_restore -U ticketing -d it_ticketing --clean --if-exists < backup_before_upgrade.dump

# 回退代码
git checkout <previous-stable-commit>

# 重新构建并启动
docker compose up -d --build

echo "✅ 回滚完成，请验证服务状态"
curl http://localhost:8000/health
```

## 6. 监控建议

### 6.1 健康检查端点

应用提供 `/health` 端点用于健康检查：

```bash
# 手动检查
curl -s http://localhost:8000/health
# 正常返回: {"status":"ok"}

# 配合 cron 定时检查
*/5 * * * * curl -fsS http://localhost:8000/health || echo "IT工单系统异常" | mail -s "健康检查失败" admin@example.com
```

### 6.2 日志监控

#### 6.2.1 日志位置

应用使用结构化日志格式，所有日志输出到 stdout/stderr，由 Docker 收集。

```bash
# 实时查看应用日志
docker compose logs -f app

# 导出日志到文件分析
docker compose logs app --since 24h > app_logs_24h.txt
```

#### 6.2.2 关键日志关键词

| 关键词 | 含义 |
|--------|------|
| `ERROR` | 错误日志，需关注 |
| `WARNING` | 警告信息 |
| `Connection refused` | 数据库连接失败 |
| `Migration` | 数据库迁移相关 |
| `CSRF validation failed` | 潜在的 CSRF 攻击 |

#### 6.2.3 日志分析脚本

```bash
# 统计最近的错误数量
docker compose logs app --since 1h | grep -c "ERROR"

# 查找 5xx 状态码
docker compose logs app --since 1h | grep -E '" [5][0-9]{2} '
```

## 7. 安全建议

### 7.1 定期更换密码

```bash
# 1. 更新 .env 文件中的以下密码：
#    - DB_PASSWORD (数据库密码)
#    - JWT_SECRET_KEY (至少 32 位随机字符串)
#    - ADMIN_DEFAULT_PASSWORD (管理员初始密码)

# 2. 生成安全的随机密码
openssl rand -base64 48 | tr -d '\n'
python3 -c "import secrets; print(secrets.token_urlsafe(48))"

# 3. 更新数据库密码后，也需要在 PostgreSQL 中更新
docker compose exec postgres psql -U ticketing -d it_ticketing -c "ALTER USER ticketing PASSWORD 'new-password';"

# 4. 重启服务
docker compose down && docker compose up -d
```

**密码更换周期建议：** 每 90 天更换一次

### 7.2 更新依赖

```bash
# 1. 查看可更新的依赖
docker compose exec app pip list --outdated

# 2. 更新 requirements.txt 中的版本号

# 3. 重新构建镜像
docker compose build --no-cache app

# 4. 测试新版本
docker compose up -d app

# 5. 如果正常，更新所有服务
docker compose up -d
```

### 7.3 防火墙配置

#### 7.3.1 推荐防火墙规则（iptables / ufw）

```bash
# 使用 UFW（Ubuntu）
ufw default deny incoming
ufw default allow outgoing

# 允许 SSH（管理用）
ufw allow 22/tcp

# 允许 HTTP/HTTPS（Web 访问）
ufw allow 80/tcp
ufw allow 443/tcp

# ⚠️ 不要直接暴露 8000 端口到公网！
# 如果必须直接暴露（仅开发环境）
# ufw allow from 192.168.1.0/24 to any port 8000

ufw enable
```

#### 7.3.2 Docker 端口安全

在 `docker-compose.yml` 中将端口绑定限制为本地：

```yaml
# 仅允许本地访问（推荐通过 Nginx 反向代理访问）
ports:
  - "127.0.0.1:8000:8000"

# 而不是：
# ports:
#   - "8000:8000"  # 直接暴露到所有网络接口
```

### 7.4 HTTPS 配置

#### 7.4.1 使用 Nginx 反向代理 + Let's Encrypt

**Nginx 配置示例（`/etc/nginx/sites-available/it-ticketing`）：**

```nginx
server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name your-domain.com;

    ssl_certificate     /etc/letsencrypt/live/your-domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/your-domain.com/privkey.pem;

    # 安全头
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=63072000" always;

    client_max_body_size 10M;  # 与 MAX_UPLOAD_SIZE_MB 保持一致

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /static/ {
        proxy_pass http://127.0.0.1:8000/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}
```

**启用 HTTPS 后，更新 `.env`：**

```bash
COOKIE_SECURE=true  # 启用 Secure Cookie
```

sudo apt install certbot python3-certbot-nginx

# 获取证书
sudo certbot --nginx -d your-domain.com

# 设置自动续期

### 7.5 其他安全建议

1. **限制数据库访问**：不将 PostgreSQL 的 5432 端口映射到宿主机
2. **定期审计用户**：检查并清理不需要的用户账户
3. **启用速率限制**：`.env` 中 `RATE_LIMIT_LOGIN_PER_MINUTE` 默认 10 次/分钟
4. **非 root 运行**：Dockerfile 已配置 `appuser` 运行，不要改为 root
5. **定期审查日志**：关注异常登录和操作记录
6. **保持系统更新**：定期更新宿主机和 Docker 引擎
